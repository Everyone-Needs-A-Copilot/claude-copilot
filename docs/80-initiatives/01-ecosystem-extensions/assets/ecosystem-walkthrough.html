How each use case exploits the ecosystem

Every workflow is the same spine — resolve across layers, act at the narrowest one — pointed at a different job. The tinted spine on each card is the tier it turns on.

UC2Daily updateOne fan-out pulls 4 layers, resolves, materialize-syncs, prints a provenance diff.

B · Authoring — one verb, at the right layer

UC5Dept-only skillSeparate dept repo — only Finance resolves it; scoped by layers.department.
UC6Company-wide skillOrg layer — every department inherits on next pull.

C · Overrides, promotion & governance

UC10Two departmentsTwo ranked dept layers — total-ordered, no ambiguity.
UC11Capability policyPer-layer allow/deny — a dept may add skills, never override the security agent.

How the hardening pass improved those use cases

Two adversarial red-teams found 13 Critical + 17 High failures. Each remediation is a use case that went from breaks in the real world to holds.

UC1Onboarding
BeforeInstaller aborts on a locked-down machine (no admin); SAML/404 silently drops the company layers — Bob thinks he has them.
AfterAdmin-free binary bundle, proxy/offline, GHES-aware; 404s classified (SSO vs not-created vs deleted) — never a silent drop.
UC2Daily update
BeforeCaret ranges + an unsynced lockfile → two machines silently run different sets. Disabled products leave orphaned content loading.
AfterResolve from the lockfile (reproducible); materialize is a reconciling sync that prunes what a removed layer left behind.
UC4Personal work
BeforeLocal-only default = new laptop wipes the accountant agent and every personal note. The least backup-savvy user, defaulted into data loss.
AfterBackup-by-default once auth exists, plus a persistent "not backed up" banner until protected.
UC8Override
BeforeYou override org qa; the org ships a security fix to it; update says "0 conflicts" and you run the vulnerable version forever.
AfterUpdate diffs shadowed layers too; a security: trailer breaks through override: true; an override-stale checker.
UC9Promotion
BeforeDeny-list scan misses a novel secret; a promoted item references a private one that didn't promote → public breakage.
AfterAllow-list scan + secret scanners + human gate; dependency-closure check promotes the whole closure or rejects.
UC11Governance
BeforeCompromised org edits away its own may_never guard; a dept renames sec.md to dodge it.
AfterPolicy signed by a security key distinct from push; classified by what content does, not its filename.
LifecycleLeaver
BeforeA departing employee keeps a permanent local copy of confidential org/dept content; revocation is server-side only.
Aftercopilot deprovision wipes on offboard; DLP posture — true secrets never materialize; confidential dept data in isolated repos.
03

From nothing → a working partner on Bob's computer

Bob is a non-technical accountant. No git, no SSH, no idea what "AI" is. He runs one thing and answers two questions a human can answer.

A bare laptop
No git, no Node, no AI tooling. Maybe no admin rights. Maybe behind a corporate proxy. The starting line for the majority of real users.
Paste one line or double-click
/bin/bash -c "$(curl -fsSL https://get.copilot.enac.dev/bootstrap.sh)" — or IT hands Bob a double-clickable offline bundle. Same code path, zero typing.
✦ Partner ready Ring 0
Installs prerequisites admin-free (no sudo) and the public foundation. Bob has a working AI partner right here — before any company, auth, or network step can fail.
Two plain questions Ring 1
"What company?" and "What team?" — suggested from his email/GitHub, persisted as config, never re-asked. A browser code signs him in (no keys). Org + Finance-department layers clone with the right credential.
Personal layer, backed up Ring 1
A private personal layer is scaffolded (and offered a backup repo so a new laptop never loses his work). Everything resolves and materialize-syncs into ~/.claude/ — focus-scoped to Finance + org + foundation only.
Verify & teach Ring 1
A self-check confirms agents load and nothing drifted, then a cheat-sheet + copilot add skill --personal — a guided way for Bob to add his own skill later without touching git or an editor.
✦ A partner that stays healthy Ring 2
Bob types claude and works. The same one command is also the repair and update tool — re-run it anytime something feels off; it self-heals and never touches his unsaved work.
04

What an IT manager sets up — once, for the whole company

The entire enterprise side is one published seed plus repos and access. After that, every employee's setup is the one command above.

IT does this once

company-wide setup

  • Publish the seed. copilot ecosystem init scaffolds <org>/copilot-ecosystem with ecosystem.yml — the one fact every employee's installer reads.
  • Declare products & pins. Which of Claude/Codex · Knowledge · CLI are enabled, and the foundation version floor.one-line edits later enable a product for everyone
  • Create the repos. Org repo + a separate repo per department (the confidentiality boundary), per product.
  • Grant team access. Finance team → Finance repos only. This is what keeps sensitive dept data isolated.
  • Sign the capability policy with a security-team key (distinct from push) — bounds what any layer may contribute.
  • Set auth & host. SSO / GitHub App for CI; GHES host params if self-hosted; the pinned foundation trust key.
  • Hand out one thing. The install one-liner, or an offline bundle for locked-down / air-gapped machines.

Then every employee

per-person, self-service

  • Runs one command (or double-clicks the bundle).
  • Answers two questions — company, team — both suggested with a smart default.
  • Gets a focus-scoped partner — only their department + org + foundation, never the whole corpus.
  • Extends when ready — copilot extend knowledge, copilot extend cli; same guided, additive flow.
The whole point: IT publishes one seed and sets access; the employee runs one command. Nobody types a repo URL, wires a manifest, or edits a config. Adding a department or enabling a product later is a one-line edit to the seed that every machine picks up on its next sync.
05

How syncing works — and how to make it run itself

Today syncing is one manual command. Everything needed to automate it already exists in the design; these are the layers we'd add so nobody has to think about it.

How one sync works (copilot update)

pull
Fan-out all 4 layers, right credential per URL. Offline → cached SHAs.
resolve
From the lockfile — reproducible; re-resolve ranges only on --upgrade.
sync
Materialize winners into ~/.claude/; prune what's no longer resolved.
diff
Provenance report — what changed, which layer won, what a shadow now hides.

Automation we could put in place

local · scheduled
Background refresh
A launchd/cron/Task-Scheduler job runs copilot update quietly (e.g. overnight). Bob's set is current every morning without a thought.
local · on-launch
Session-start freshness
When claude/copilot starts, check staleness against a threshold and sync only if overdue — fresh without pulling on every launch.
notify · push
Publish → propagate
An org publish fires a webhook (or clients cheaply poll a single "latest-lockfile-SHA" endpoint), so a company update lands in hours, not whenever someone remembers.
org · policy
Cadence in the seed
ecosystem.yml sets update policy: auto-pull foundation patches, hold majors for approval — the org tunes velocity vs. stability centrally.
server · CI
Shift-left validation
CI on each layer repo validates requires vs the floor, the capability policy, that declared dept repos exist, and runs the promotion leak-scan — bad content never reaches a client.
local · self-heal
Doctor on a timer
copilot doctor runs periodically to detect drift (a hand-edited file, a moved repo, expired auth) and repair it — the same idempotent engine as install.
The arc: manual copilot update today → a background, push-aware, self-healing sync you never think about — with the org setting how fast changes roll and CI guaranteeing only valid content is ever published. Every piece rides machinery the architecture already defines (lockfile, reconciling sync, doctor, ecosystem.yml); none of it is net-new invention.
06

What this is still missing

The honest list — beyond the red-team fixes. Chips mark status: P2 planned-hard · unbuilt no design yet · open needs a decision · scoped-out deliberately deferred.

Observabilityopen
No telemetry on what actually gets used, corpus health, or adoption. You can't improve descriptions/routing you can't measure.
Protocol compositionP2
Protocol still resolves whole-unit override; the "a department inserts a stage into the chain" composer is designed-hard, not built.
CLI-integration scopingunbuilt
The seam that lets connectors layer company-vs-personal is net-new — no existing mechanism in CLI Copilot yet.
Knowledge search qualityopen
FTS5 keyword only across 4 corpora — no semantic ranking, cross-layer dedup, or freshness weighting. "Focus" bounds it; relevance isn't solved.
Author onboardingopen
Bob's consumer flow is designed end-to-end; how a dept lead (Mira) learns to author, review, and promote is not.
Conflict-resolution UXopen
When a personal override genuinely conflicts with a changed upstream, the merge/repair experience for a non-developer is undefined.
Non-GitHub SCMscoped-out
The whole model assumes GitHub. GitLab / Bitbucket / Azure DevOps enterprises are unaddressed in v1.
Multi-org / consultantscoped-out
A user in two orgs breaks single-tenant assumptions. The namespace seam is pre-built; the feature is deferred.
Cloud Copilotscoped-out
The fourth named project has no repo or definition. Vision-only until scoped.
Compliance / residencyopen
Materialized confidential copies land on laptops. Where they may legally live (GDPR/data-residency) needs an explicit policy.
Cost & scaleopen
Big orgs, many layers, large corpora — sync fan-out, rate limits, and materialize cost at thousands of employees are unmodeled.
Non-terminal surfacesscoped-out
Everything assumes a shell. A truly non-technical user on a managed Chromebook / web-only device has no path.
