**Reviewer stance:** find where it breaks when real people use it. No praise.
**Docs under test:** `02-four-tier-and-github-topology.md`, `03-use-cases.md`, `design-naming-topology.md`, `design-bootstrap-orchestration.md`, `design-product-composition.md`.

## Findings (Critical / High first)

| # | Scenario | Failure (concrete, step-level) | Sev | Root cause | Proposed fix |
|---|---|---|---|---|---|
| C1 | UC1 — locked-down macOS, no admin | `bootstrap.sh` P0 `ensure_pkg_mgr()` runs the Homebrew installer, which writes to `/opt/homebrew` and **shells out to `sudo`**. On MDM-managed corporate macOS Bob has no admin rights → Homebrew install exits non-zero → `set -euo pipefail` aborts the *entire* script at P0. Bob ends with **nothing** — not even the foundation-only fallback the design promises ("failure never leaves him with nothing"). | **Critical** | Ring 0 assumes it can install system packages; the "you always keep the foundation" invariant is violated *before* the foundation clone (P1) is ever reached. | Ship prebuilt, admin-free binaries: a userland tarball of `git`(if absent)/`gh`/`node`/`cc`/`tc`/`copilot` unpacked into `~/.copilot/bin` (no brew, no sudo). Detect no-admin and take the tarball path. Never let a prereq failure abort before P1. |
| C2 | UC1 — bootstrap handoff | The stage-0 sketch's last line is `exec copilot doctor --bootstrap`. But P0 only installs `git gh node` + `cc` + `tc` (via their `install.sh`). **Nothing installs the `copilot` binary or puts it on PATH.** `have copilot` is never checked. The `exec` fails with `command not found`, dropping Bob at a dead prompt right after "Foundation ready." | **Critical** | The three-ring model names `copilot doctor` as Ring 1 "shipped in the foundation" but the bootstrap never installs/links it. cc & tc have install steps; `copilot` has none. | Add `have copilot || bash "$LAYERS/foundation/tools/copilot/install.sh"` (or symlink into `~/.copilot/bin`, added to PATH) *before* the `exec`. Verify `command -v copilot` and fail loud with a fix hint if missing. |
| C3 | UC1 — proxy / SSL-inspection firewall | `curl -fsSL https://get.copilot.enac.dev/bootstrap.sh` is a **redirect to `raw.githubusercontent.com`**. Corporate proxies very commonly allowlist `github.com` but **block `raw.githubusercontent.com`** (and `objects.githubusercontent.com` for release assets). Ring 0 never downloads → the one-liner prints nothing or a TLS/404 error → Bob is stuck at step one with no diagnosis. SSL-inspection also breaks `curl -f` cert validation unless the corp root CA is trusted for that exact chain. | **Critical** | Single fragile fetch path through a host enterprises routinely block; no proxy-awareness, no offline bundle. | Host `bootstrap.sh` on a domain that serves the bytes directly (not a raw.githubusercontent redirect). Provide (a) a proxy-aware mode honoring `HTTPS_PROXY`/`NO_PROXY`, and (b) an **air-gapped install bundle** (`.command`/`.tar`) IT can hand out that carries the foundation + binaries with no network fetch. |
| C4 | UC1 — Windows | The **entire installer is POSIX bash** (`bootstrap.sh`) plus a macOS-only `.command` double-click fallback. Windows has no bash on `cmd`/PowerShell by default. Claude Code runs on Windows, so real Bobs are on Windows — and there is **no Windows code path at all**: no `.ps1`, no winget/scoop prereq install, no path handling for `~/.copilot` vs `%USERPROFILE%`. | **Critical** | Design scoped to macOS/Linux only; Windows non-developers (the majority of corporate accountants) are unaddressed. | Author a parallel `bootstrap.ps1` (winget/scoop prereqs, `gh` device flow, same Ring-1 handoff) and normalize all path logic. Or gate: explicitly declare Windows = WSL-only and detect+guide WSL install. |
| C5 | UC1 — GitHub Enterprise Server (github.acme.com) | The naming convention and SSH aliases **hardcode `HostName github.com`**, and the foundation is a constant `https://github.com/Everyone-Needs-A-Copilot/...`. A GHES-only enterprise (a) routes all private repos through `github.acme.com`, not `github.com`, and (b) frequently **firewalls public github.com entirely**. So the `github-work` alias points at the wrong host *and* the anon-HTTPS foundation clone is unreachable. `gh` also needs `--hostname github.acme.com`. `ecosystem.yml` discovery (`git@github-work:<org>/copilot-ecosystem`) assumes github.com. GHES is effectively unsupported. | **Critical** | Every host reference is a literal `github.com`; no `host`/`api_base` parameter in `ecosystem.yml` or the SSH-alias generator. | Add `host`/`api_base`/`ssh_host` to `ecosystem.yml`; make the SSH-alias `HostName` and the foundation URL parametric. Provide a foundation **mirror path** for enterprises that block public github.com (self-hosted foundation mirror declared in `ecosystem.yml`). |
| C6 | UC4 — local-only personal, new laptop | Q4 defaults to **No** (local-only personal layer, no remote). Bob authors `accountant.md` via `copilot add skill --personal`; the doc says it "leaves it local." Bob gets a new laptop (or disk dies). Bootstrap scaffolds a **fresh empty** personal layer. **The accountant agent — and every personal skill/note Bob ever made — is permanently gone.** The persona least likely to have git/backup hygiene is defaulted into the data-loss option, with no "this is not backed up" warning. | **Critical** | The safe-by-default choice (no remote) is also the lose-everything choice for a non-technical user; the never-destroy invariant protects the working tree from *doctor*, not from hardware loss. | Default Q4 toward a **backed-up** personal layer (auto-create a private repo via `gh repo create` once auth exists), or at minimum surface a persistent "your personal work is NOT backed up — run `copilot personal publish` to protect it" banner on every `add skill --personal` and every `doctor` run. |
| H7 | UC1 — SSO/SAML-gated org | `gh auth login` device flow mints an OAuth token, but for a SAML-protected org the token/SSH key must be **separately SSO-authorized in a browser** before it can read org repos. Until then, GitHub returns **404 (not 403)** for those private repos to avoid leaking existence. The bootstrap's `layer-present`/404 handler "drops it from the effective set, warns" — so Bob **silently loses org + dept layers** and is told the repo is "gone," when it exists and he simply hasn't SSO-authorized. Same trap on the developer path: generated SSH keys aren't SSO-authorized. | **High** | 404-means-deleted assumption collides with SAML's 404-means-not-authorized behavior; no SSO-authorization step in either auth path. | Distinguish 404-after-auth from 404-unauthorized (probe `gh api /orgs/<org>` / check org SSO state). On SAML orgs, drive the `gh auth login --web` SSO authorization and, for SSH, print the "Configure SSO" URL for the uploaded key. Never classify a SAML 404 as "layer gone." |
| H8 | UC2 — two machines, different SHAs | Foundation ref is a **semver range** (`^5.14.0`), not a pin. Machine A updates before 5.14.1 ships → resolves 5.14.0; machine B updates after → resolves 5.14.1. `copilot.lock` is the reproducibility anchor, but it lives in the personal layer — and for Bob (Q4=No, no remote) **the lockfile never syncs between machines**. The two laptops silently run different resolved sets, directly contradicting UC3/UC12's "reproducible on any machine" claim. | **High** | Caret ranges + a per-machine, un-synced lockfile = guaranteed drift; the reproducibility guarantee has no transport for the artifact that provides it. | Commit `copilot.lock` to a synced location (the personal remote, or a project repo). `copilot update` should resolve *from the lockfile* by default and only re-resolve ranges on an explicit `--upgrade`. Warn when running range-resolution against an unsynced lockfile. |
| H9 | UC2 — foundation MAJOR bump under pins | `ecosystem.yml` floors foundation at `^5.14.0` (caret excludes 6.0 — good). But an org/dept layer authored against 6.0 carries `requires: minVersion 6.0` (doc 02 §8 / UC12). On update the resolver **hard-errors** because the pinned foundation (≤5.x) can't satisfy the dept's `requires`. A single dept push authored against a newer foundation **breaks `copilot update` for every consumer**, including Bob, who did nothing. | **High** | Cross-layer `requires` can demand a foundation newer than the org's own pinned floor, with no pre-merge check; the failure surfaces at every consumer, not the author. | Validate `requires` against the org's pinned foundation floor **at author/CI time** in the dept/org repo (reject the push). At consume time, degrade: drop the unsatisfiable layer item with a clear message, never hard-fail the whole update. |
| H10 | UC2 — dept slug renamed in ecosystem.yml | IT renames `finance`→`fin` in `ecosystem.yml`. Bob's `cc config layers.department=finance` is "the contract" and is **not reconciled** against the new `departments[]`. Next `copilot derive` templates `copilot-claude-dept-finance` → **404** → `layer-moved`/404 handler drops it and warns. Bob **silently loses his whole department layer**; the "stored config is the contract" rule guarantees the stale value wins over the fresh authoritative list. | **High** | Two sources of truth for the department (local cc config vs `ecosystem.yml departments[]`) with no drift check between them. | On `derive`, cross-check `layers.department` against `ecosystem.yml departments[]`; if the stored slug is absent, **prompt to remap** (offer the closest match) instead of computing a 404 and silently dropping. Support a `renamed_from:` alias field in `ecosystem.yml`. |
| H11 | UC2 — layer force-pushed / rebased | Org/dept/foundation are **read-only mirrors** Bob never commits to. When upstream force-pushes/rebases, `git pull --ff-only` fails. The `layer-fresh` repair says "if diverged, **stash-and-flag, never discard**" — but there is nothing of Bob's to stash; the divergence is entirely upstream. Doctor gets **stuck flagging a phantom divergence** on a layer it explicitly "owns as a read-only mirror it can always re-clone," blocking the update. | **High** | `layer-fresh` conflates the personal-layer policy (never discard) with the read-only-mirror policy (reset/reclone freely); one repair action for two opposite ownership models. | Split the repair by layer role: read-only mirrors → `git fetch && git reset --hard <tip>` or reclone on non-ff; personal layer → the stash-and-flag guard. Key the repair on `role`, not one global rule. |
| H12 | UC5 — declared-but-uncreated dept repo | `ecosystem.yml departments: [finance, engineering, platform]` but IT never actually created `copilot-claude-dept-platform` on GitHub. A Platform user's `derive` computes the URL, clone **404s**, checker drops it and warns. The user gets org+foundation only while believing they have Platform content. The "closed `departments[]` set prevents 404s" claim only blocks *unknown-slug* 404s — it does **not** verify the declared repos exist. | **High** | `departments[]` is trusted as ground truth for existence but is only a naming list; no existence verification at publish or derive time. | `copilot derive`/`doctor` should `gh repo view` each declared layer repo and **hard-warn "declared in ecosystem.yml but not found on GitHub"** — distinct from "layer deleted." Add a CI lint on `copilot-ecosystem` that verifies every declared dept repo exists. |
| H13 | UC6 — org↔foundation silent name collision | Raj publishes org `skills/excel-to-json`. Foundation later ships its own `excel-to-json`. Rank 30 (org) < 40 (foundation) → org **silently shadows** foundation for the whole company, permanently. If Bob's workflow depended on the foundation version's behavior, it changes with **no warning to Bob** (he authored nothing; UC8's "shadow is reported" only covers a user's *own* personal override). Foundation improvements to that skill never reach anyone while the org copy shadows it — a stale-shadow capability hijack. Raj also gets **no author-time signal** he collided with a foundation name (optional npm-scope namespacing is not enforced). | **High** | Override semantics + no mandatory namespace/collision guard = an upper layer can silently capture a lower-layer capability name for all consumers, with no notice to author or consumer. | Enforce namespacing (`acme/excel-to-json` vs `foundation/excel-to-json`) OR require an explicit `shadows: foundation/excel-to-json` acknowledgment at author time (CI-checked). Surface org-over-foundation shadows in the `copilot update` diff prominently, not just in `resolve --explain` nobody runs. |
| H14 | UC4 — materialize-drift clobbers Bob's edits | The never-destroy guard protects the personal **layer** working tree. But Bob's mental model (per the P10 cheat-sheet: "Talk to it → claude") is that he edits the files claude uses — which are the **materialized `.claude/` copies**, "derived, safe to overwrite." When Bob tweaks `.claude/agents/accountant.md` directly and later runs `copilot repair`, `materialize-drift` **silently overwrites his edits** from the layer source. The guard covers the wrong tree relative to where a non-dev actually edits. | **High** | The safety invariant assumes the user edits the layer (source of truth); the taught workflow leads the user to edit the materialization (disposable). | On `materialize-drift`, detect that the drifted file diverged from source and **offer to fold the edit back into the personal layer** (or refuse + warn) rather than silently clobbering. Teach editing via `copilot add/edit skill`, and make `.claude/` materialized files read-only where the OS allows. |
| H15 | UC1 — which `.claude/` for a project-less Bob | Materialize-by-copy targets `.claude/`, which is **per-project/per-cwd** in Claude Code. Bob is a non-developer with **no project repo**. Where does materialization land — `~/.claude/`? The cwd where he happened to run the one-liner (e.g. `~/Downloads`)? If he later runs `claude` from a different directory, does his materialized set follow him? The entire tiered model is project-scoped, but the governing persona has no project. Undefined. | **High** | The resolver/materialize target is defined for a developer-in-a-repo; the non-dev "just run claude" flow has no defined materialization root. | Define an explicit **user-global materialization root** (`~/.claude/`) for the Bob/solo path, independent of cwd, and document precedence when Bob is *also* inside a project. Make `copilot doctor` assert the root exists and is on the Claude Code discovery path. |
| M16 | UC1 — org hasn't published ecosystem.yml | Discovery chain (seed repo → `.github` → topic) all miss. `copilot derive` has no input: it can't know enabled products, departments, auth, or pins. Behavior is **undefined** — best case Bob degrades to foundation-only, but the UC1 promise ("company's agents/skills in one command") silently fails with no clear "your org hasn't set this up yet — here's what IT must publish" message. | **Med** | No specified degrade path or actionable error when the single published seed is absent. | Detect all-three-miss and print an explicit "no `copilot-ecosystem` found for `<org>` — foundation-only mode; give IT this template" message. Provide IT a `copilot ecosystem init` scaffold generator. |
| M17 | UC2 — laptop offline | `copilot update` pulls 4 layers; offline → all pulls fail with **network-unreachable**, which is distinct from divergence. It's unclear whether the checkers treat unreachable as `fail` (score drop, exit 1, block materialize) or as skip-and-materialize-from-cache. If it blocks, offline Bob can't even re-materialize from already-cloned layers — a regression from "foundation always works." | **Med** | Network-unreachable is not a declared checker state; conflatable with divergence/failure. | Treat unreachable as **"offline — using cached SHAs"**, skip the pull, and still materialize from the local clones. Reserve `fail` for genuine drift, not connectivity. |
| M18 | UC5 — author gets no policy-denial feedback | Mira adds `agents/sec.md` or an `mcp` entry to dept-finance and pushes successfully (git accepts it; capability policy is enforced at **consumers' materialize time**). The `POLICY … DENIED` line prints on **every consumer's** machine, never Mira's. She believes it shipped; nobody receives it; she has no feedback loop to learn her contribution was structurally dropped. | **Med** | Capability policy enforced only at the consume edge, not at author/push time. | Mirror the capability policy as a **CI check / pre-receive hook** on the dept/org repo so a denied contribution is rejected at push with the exact `may_never` reason — fail at the author, not silently at 500 consumers. |
| M19 | UC1 — anon github.com rate limits (enterprise rollout) | Foundation clone + `gh api` discovery are **unauthenticated** (60 req/hr per IP for API; clone throttling). A whole department onboarding the same morning from one office **behind shared NAT** presents as a single IP → anon rate limits trip → foundation clones and `.github`/topic discovery **fail intermittently** with opaque errors, exactly during a coordinated rollout. | **Med** | Anon-first for the public floor ignores shared-egress rate limiting at enterprise scale. | Use the already-present `gh` token for discovery API calls once auth exists; add retry/backoff on 429; document/support an internal foundation mirror for large rollouts. |
| M20 | UC1 — authenticated/transparent proxy for git | Even if `curl` succeeds, `git clone` (anon HTTPS + SSH) does **not** inherit an authenticated corporate proxy unless `http.proxy`/`GIT_PROXY` is set with credentials; SSH (port 22) is frequently blocked outbound entirely, breaking the `github-work` SSH-alias clones. Clones hang (no `BatchMode` on the HTTPS path) or fail. | **Med** | No proxy configuration or SSH-over-443 fallback in the clone paths. | Detect proxy env and configure `git http.proxy`; offer SSH-over-HTTPS (`ssh.github.com:443` / `HostName ssh.github.com Port 443`) when 22 is blocked; set `GIT_SSH_COMMAND` with `BatchMode=yes` to fail fast, not hang. |
| M21 | Cross-cutting — `copilot` name collision | The binary is literally `copilot`. GitHub ships **`gh copilot`** and there is a standalone GitHub `copilot` CLI; the user's own environment already aliases `copilot` (per CLAUDE.md: "the `copilot` name is a shell alias"). On a corporate machine with GitHub Copilot CLI installed, `exec copilot …` may resolve to the **wrong binary**, and `copilot doctor` collides semantically. | **Med** | Chose a heavily-contested command name with no namespacing. | Use a distinct verb (`enac`, `ccx`, or invoke via `cc copilot …`) or resolve by absolute path (`~/.copilot/bin/copilot`) in bootstrap; detect a conflicting `copilot` on PATH and warn. |
| M22 | UC3 — provenance is reconstructed, not trusted for edited files | `resolve --explain` reads `copilot.audit.json` (source SHA, signer captured at materialize). If a materialized `.claude/` file is **hand-edited** afterward, explain still prints "SHA a1b2c3d, signed ✓" — a **false provenance** for a file that no longer matches. Only `doctor materialize-drift` catches it, and only if run. Also "signed ✓" is **commit-tip** granularity, not per-file: a file added by an unsigned earlier commit under a signed tip is reported as signed. | **Med** | Explain trusts recorded metadata without re-hashing the live file; signature attribution is coarser than per-item. | `resolve --explain` should hash the live file and flag `MODIFIED — no longer matches recorded SHA`. Record per-file introducing-commit signer, not just tip signer. |
| L23 | UC1 — `get.copilot.enac.dev` single point of failure | The one-liner depends on an ENAC-controlled domain that redirects to raw.githubusercontent. DNS lapse, cert expiry, or ENAC domain loss = **nobody on earth can install**, and it's a second dependency stacked on the github redirect. | **Low** | Install entry point centralized on one owner-controlled hostname. | Publish the one-liner as a direct github.com raw URL fallback; document the bare `git clone` install path so the ecosystem survives the vanity domain lapsing. |
| L24 | UC9 — deny-list leak scan is incomplete | Promotion's "hard-fail leak scan" is a **deny-list** (client names, `.env`, tokens, `mcp` secrets). Deny-lists inherently miss novel secret shapes / renamed client references, so private content **can leak to the public foundation**. (ENAC-side; lower Bob impact.) | **Low** | Deny-list is a blocklist, not an allowlist. | Add entropy/secret-scanning (gitleaks-style) + an allowlist of promotable paths; require human sign-off on the GitHub Environment gate (already present) as the real backstop. |

## Top 5 must-fix before this ships

1. **C1 + C2 + C3 together = Bob's happy path does not execute on a real corporate machine.** No-admin aborts before the foundation, the `copilot` binary the whole Ring-1 handoff depends on is never installed, and the download host is one enterprises routinely block. Fix: admin-free binary bundle + install `copilot` on PATH + proxy-aware/offline install bundle. This is the single most damaging cluster — it defeats the governing test (UC1) at step one.
2. **C6 — local-only personal default is a data-loss trap** for exactly the non-technical persona the design centers on. Flip the backup default or blare a persistent "not backed up" warning.
3. **H7 + H10 + H12 — the "404 = layer gone, drop and warn" handler silently strips company layers** on SAML orgs, after slug renames, and for declared-but-uncreated repos. Bob thinks he has company content and doesn't. Classify 404s correctly and never silently degrade below the promised set.
4. **H8 + H9 — reproducibility is unbacked and version-skew can hard-brick update.** Sync the lockfile and resolve from it; validate cross-layer `requires` against the pinned floor at author time so one dept push can't break every consumer.
5. **C4 / C5 — Windows and GitHub Enterprise Server are entire unsupported populations.** Most corporate accountants are on Windows; most large enterprises are on GHES with public github.com firewalled. Either support them or explicitly scope them out — silently assuming macOS+github.com will strand the majority of the target market.
